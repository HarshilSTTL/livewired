### [2026-04-01 10:00] | FIX | get_profile_events — recurring events expand across weeks (dynamic approach, later superseded)

**Root cause:** `get_profile_events` was querying `event_mst` by exact `event_date` only. When a recurring event was created, only one row existed in `event_mst`. The SP never read `event_recurring`, so recurring events only appeared on their original date and were invisible on subsequent weeks.

**Fix (approach 1 — later replaced):** Complete rewrite using two branches united with UNION ALL:
1. Non-recurring branch — unchanged date range filter
2. Recurring branch — uses `generate_series` to enumerate all 7 days in the window, cross-joins with `event_mst JOIN event_recurring`, applies recurrence math per type

> ⚠️ This approach was replaced at [2026-04-01 11:00] with the pre-generation model.

**Files changed:**
- `functions/events/get_profile_events.md` — dynamic expansion rewrite
- `docs/api/events/get_profile_events.md` — updated logic flow docs

### [2026-04-01 11:00] | FIX + SCHEMA | create_event + get_profile_events — switch to pre-generated occurrence rows

**New approach:** At user request, dynamic expansion replaced with pre-generation. When a recurring event is created, individual rows are inserted into `event_mst` for every occurrence date — no runtime expansion needed.

**Schema change:** `event_mst` gets new column `parent_event_id uuid NULL REFERENCES event_mst(event_id) ON DELETE CASCADE`
- Parent/template row: `parent_event_id = NULL`, `is_recurring = true`
- Child occurrence rows: `parent_event_id = <parent uuid>`, `is_recurring = true`, each with their specific `event_date`
- ON DELETE CASCADE — deleting parent removes all children automatically

**create_event changes:**
- After inserting parent row + event_platforms + event_recurring, loops through all occurrence dates
- `weekly` type: per day, find first occurrence on/after start_date, step +7×interval days until end_date
- `first` type: per day, per month — find first occurrence of that weekday in the month
- `last` type: per day, per month — find last occurrence of that weekday in the month
- If recurring_end_date is null, generates up to 1 year from start_date
- event_platforms stored on parent only — children inherit via COALESCE in get_profile_events

**get_profile_events changes:**
- Simplified back to plain date-range query
- Filter: `WHERE event_date BETWEEN ... AND (is_recurring = false OR parent_event_id IS NOT NULL)`
- Platform subquery: `WHERE ep.event_id = COALESCE(e.parent_event_id, e.event_id)`

**Files changed:**
- `schema/tables/08_event_mst.md` — added parent_event_id column + comments
- `docs/database/tables/08_event_mst.md` — documented parent/child row model with example table
- `functions/events/create_event.md` — added occurrence generation loop (weekly + first + last)
- `functions/events/get_profile_events.md` — simplified back to plain date-range query
- `docs/api/events/create_event.md` — documented pre-generation, algorithm, example table

### [2026-04-01 14:00] | TABLE + SP | Google OAuth support — auth_provider column + google_auth SP

**Table change:** `users` — added `auth_provider text DEFAULT 'email'`
- Existing users automatically get `auth_provider = 'email'`
- Google users get `auth_provider = 'google'`, `password = NULL`

**New SP:** `google_auth` (POST /rpc/google_auth)
- Input: `p_email text` (from Supabase OAuth session)
- If email exists → return existing user_id (login)
- If email not found → INSERT new user, return new user_id (signup)
- Handles both signup + login in one call
- `register` and `login` SPs unchanged — still used for email/password flow

**Migration required:**
