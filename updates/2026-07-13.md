### [2026-07-13] | FEAT + SCHEMA | Migrate auth to Supabase Auth — email verification implemented

**What changed:**
Replaced custom password-based auth (register/login/forgot_password RPC functions) with **Supabase Auth** (`auth.users`). Email verification is now handled natively by Supabase — a verification email is sent automatically on signup, and the user's account is marked verified when they click the link.

**Why:**
Custom auth stored plain-text passwords in `public.users.password` with no token/session management. Switching to Supabase Auth gives us secure hashed passwords, JWT sessions, built-in email verification, and password reset — without maintaining that logic ourselves.

---

#### Schema changes — `public.users`

```sql
-- Removed password column (Supabase Auth manages passwords)
ALTER TABLE public.users DROP COLUMN IF EXISTS password;

-- Added email verification flag (synced via trigger from auth.users)
ALTER TABLE public.users
    ADD COLUMN IF NOT EXISTS is_email_verified boolean NOT NULL DEFAULT false;
```

**Note:** A FK from `public.users.id → auth.users.id` was considered but skipped — existing rows were generated independently and would fail the constraint check. The relationship is enforced going forward by the trigger below.

---

#### New trigger 1 — auto-create `public.users` row on signup

Fires `AFTER INSERT ON auth.users`. When frontend calls `supabase.auth.signUp()`, Supabase creates the `auth.users` row, and this trigger immediately creates the matching `public.users` row using the same UUID.

```sql
CREATE OR REPLACE FUNCTION public.handle_new_auth_user()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
BEGIN
    INSERT INTO public.users (id, email, username, auth_provider, created_at, updated_at)
    VALUES (
        NEW.id,
        NEW.email,
        NEW.raw_user_meta_data->>'username',
        COALESCE(NEW.raw_app_meta_data->>'provider', 'email'),
        now(), now()
    )
    ON CONFLICT (id) DO NOTHING;
    RETURN NEW;
END;
$$;

CREATE OR REPLACE TRIGGER on_auth_user_created
    AFTER INSERT ON auth.users
    FOR EACH ROW EXECUTE FUNCTION public.handle_new_auth_user();
```

---

#### New trigger 2 — sync `is_email_verified` when user clicks verification link

Fires `AFTER UPDATE ON auth.users`. When `email_confirmed_at` transitions from NULL to a timestamp (user clicked the link), this trigger sets `is_email_verified = true` in `public.users`.

```sql
CREATE OR REPLACE FUNCTION public.handle_email_verified()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER SET search_path = public AS $$
BEGIN
    IF OLD.email_confirmed_at IS NULL AND NEW.email_confirmed_at IS NOT NULL THEN
        UPDATE public.users
        SET is_email_verified = true, updated_at = now()
        WHERE id = NEW.id;
    END IF;
    RETURN NEW;
END;
$$;

CREATE OR REPLACE TRIGGER on_auth_email_confirmed
    AFTER UPDATE ON auth.users
    FOR EACH ROW EXECUTE FUNCTION public.handle_email_verified();
```

### [2026-07-13] | FEAT + SP + SCHEMA | Multiple event reminders per user per event — implemented

**Status:** ✅ Implemented (supersedes the plan below — two details changed during implementation, noted inline).

**What changed:**
- `event_reminders` UNIQUE constraint changed from `(user_id, event_id)` → `(user_id, event_id, reminder_minutes)`, allowing multiple reminder times per user per event.
- New SP `set_event_reminder` (final name — plan below said `set_event_reminder` too, so no rename needed) — adds one reminder time. **Unlimited reminders per event** (plan's original answer), not the capped version I built on the first pass.
- New SP `delete_event_reminder` — soft-deletes one reminder row by `reminder_id`, scoped to the caller.
- `get_event_reminder` updated to return a `reminders: [...]` array (breaking change — old single `reminder_minutes` shape removed).
- `process_event_reminders` — comment-only update; no logic change needed, it already fires per-row.
- **Recurring propagation implemented:** `set_event_reminder` resolves the series root (`COALESCE(parent_event_id, event_id)`) and, if the event is part of a recurring series, applies the reminder to every occurrence with `event_date >= CURRENT_DATE`. Non-recurring events only get the single reminder.

**Known asymmetry (flagged, not yet resolved):** `set_event_reminder` bulk-applies across future occurrences, but `delete_event_reminder` only removes one occurrence's row at a time (`reminder_id` is per-occurrence). There is no bulk-delete-across-series counterpart yet — removing a series-wide reminder currently requires calling delete once per occurrence.

### [2026-07-13] | FEAT | Plan: Multiple event reminders per user per event #enhancement (original plan — see implemented entry above)

**Status:** Design finalized — implementation pending.

**Business rule changed:**
A user can now set **multiple reminder notifications** for a single event (e.g. 5 min before, 2 hours before, 1 day before). Previously limited to one reminder per user per event.

**Answers confirmed:**
- Reminder input: both presets and free custom minutes input
- Max reminders: unlimited
- Delete: individual reminders can be deleted by their own `id`
- Each reminder has its own `is_notified` flag (already per-row in table)
- Recurring events: setting a reminder on any child occurrence applies to **all future occurrences** of the series (`event_date >= CURRENT_DATE`)
- Existing data: kept as-is

**Planned schema change — `event_reminders`:**

```sql
-- Drop old single-reminder constraint
ALTER TABLE event_reminders
    DROP CONSTRAINT event_reminders_user_id_event_id_key;

-- Add new constraint: same user can't set duplicate times for same event
ALTER TABLE event_reminders
    ADD CONSTRAINT event_reminders_user_event_minutes_key
    UNIQUE (user_id, event_id, reminder_minutes);
```

**Planned new/updated functions:**

| Function | Type | Description |
|---|---|---|
| `set_event_reminder` | New SP | Add one reminder (user + event + minutes). Loops all future child occurrences if recurring. Reactivates soft-deleted row if same minutes exists. |
| `delete_event_reminder` | New SP | Soft-delete a single reminder row by `id`. |
| `get_event_reminder` → v2 | Updated SP | Returns array of all active reminders `[{id, reminder_minutes, is_notified}]` instead of single value. |
| `process_event_reminders` | No change | Already fires per-row — works correctly with multiple rows. |
