### [2026-06-02] | FIX | Add p_parent_event_id to exclude recurring series from conflict check #critical

**Problem:**
When editing a recurring event occurrence and submitting without any changes, the system showed a false conflict error because the event was checking itself against all occurrences in the same recurring series.

**Root Cause:**
`check_event_conflict` had no way to exclude an entire recurring series — it could only exclude a single event by `p_event_id`, but all sibling occurrences (same `parent_event_id`) were still being checked.

**Solution:**
Added `p_parent_event_id uuid DEFAULT NULL` parameter.
WHERE clause updated in both COUNT and SELECT queries:
```sql
AND (p_parent_event_id IS NULL OR parent_event_id != p_parent_event_id)
```

**Usage:**
```json
// Non-recurring event (exclude only this event)
{ "p_event_id": "abc-123" }

// Recurring occurrence (exclude this event + entire series)
{ "p_event_id": "abc-123-mon", "p_parent_event_id": "abc-123-parent" }

// Creating new event (no exclusion needed)
{ }
```

- [[functions/events/check_event_conflict.md]] — Added parameter + WHERE clause (lines 66, 114, 140)
- [[docs/api/events/check_event_conflict.md]] — Added 3 request examples with explanation

### [2026-06-02] | FIX | Implement correct overlap detection with proper boundary handling #critical

**Problem:**
- `6:00 PM (no end) vs 1:00–6:00 PM` was incorrectly showing as **CONFLICT**
- `1:00 PM (no end) vs 1:00–6:00 PM` was incorrectly showing as **NO CONFLICT**

**Root Cause:**
Using strict operators (`<`, `>`) on both paths caused boundary cases to fail. Then switching fully to inclusive (`<=`, `>=`) made adjacent events conflict.

**Solution:**
Implemented CASE logic with different boundary rules per scenario:

| Scenario | Logic | Reason |
|----------|-------|--------|
| Point-in-time (no end_time) | `existing_start <= point AND existing_end >= point` | Inclusive — time falling exactly on a boundary is a conflict |
| Range (with end_time) | `existing_start < new_end AND existing_end > new_start` | Exclusive — adjacent events are NOT conflicts |

**SQL Implementation:**
```sql
CASE 
  WHEN p_event_end_time IS NULL THEN
    -- Point-in-time: inclusive check
    (event_date || ' ' || event_time)::timestamp <= v_new_start
    AND (event_date || ' ' || event_end_time)::timestamp >= v_new_start
  ELSE
    -- Range: exclusive check (allows adjacency)
    (event_date || ' ' || event_time)::timestamp < v_new_end
    AND (event_date || ' ' || event_end_time)::timestamp > v_new_start
END
```

**Verified Test Cases:**

| New Event | Existing Event | Expected | Result |
|-----------|---------------|----------|--------|
| 1:00 PM (no end) | 1:00–6:00 PM | ❌ CONFLICT | ✅ Pass |
| 2:30 PM (no end) | 1:00–6:00 PM | ❌ CONFLICT | ✅ Pass |
| 6:00 PM (no end) | 1:00–6:00 PM | ✅ NO CONFLICT | ✅ Pass |
| 12:30–14:00 | 13:00–18:00 | ❌ CONFLICT | ✅ Pass |
| 16:00–20:00 | 13:00–18:00 | ❌ CONFLICT | ✅ Pass |
| 6:00–7:00 PM | 1:00–6:00 PM | ✅ NO CONFLICT | ✅ Pass |
