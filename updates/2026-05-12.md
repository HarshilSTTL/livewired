### [2026-05-12 13:00] | FIX | `update_follow_event_subscription` — allow any minutes (1-1440)

**What changed:**
Updated notification minutes validation to accept any value from 1 to 1440 minutes instead of only preset values (5, 10, 15).

**Root cause:**
Initial implementation restricted to only 3 preset options (5, 10, 15 min), but users need flexibility to set custom intervals like 3, 14, 21, etc.

**Fix:**
Changed validation from `IN (5, 10, 15)` to range check `BETWEEN 1 AND 1440`. Users can now set any minute value from 1 minute to 24 hours.

**Valid ranges:**
- `p_event_notification_minutes = 1-1440` → any minutes before event (1 min = 1 min before, 1440 = 24 hours before)
- `p_event_notification_minutes = NULL` → at event start time

### [2026-05-12 12:00] | SCHEMA | `event_reminders` — add soft delete columns

**What changed:**
Added soft delete columns to `event_reminders` table to support soft-delete pattern consistently across all tables.

**What added:**
- `is_deleted` (boolean, default false) — soft delete flag
- `deleted_at` (timestamptz, nullable) — timestamp of soft delete

**Migration required:**
```sql
ALTER TABLE public.event_reminders
  ADD COLUMN IF NOT EXISTS is_deleted boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS deleted_at timestamptz NULL;
```

**Soft delete pattern:**
- Set `is_deleted = true` and `deleted_at = now()` instead of deleting rows
- All queries must filter `is_deleted = false` to exclude deleted reminders
- Maintains audit trail and data integrity

### [2026-05-12 10:00] | FEAT + TABLE + DOCS + CRON | Profile-level event subscriptions for followers

**What changed:**
Followers can now subscribe to ALL events on profiles they follow. Replaced profile owner notification system with follower-based subscriptions. Followers configure when to receive notifications: before event (5/10/15 min) or exactly at start.

**What added:**
- Two new columns to `follows` table: `event_notification_enabled` (boolean), `event_notification_minutes` (int nullable)
- New SP `update_follow_event_subscription(p_user_id, p_profile_id, p_event_notification_enabled, p_event_notification_minutes)`
- New table `follow_event_subscription_dispatches` for exactly-once delivery guarantee (composite PK: user_id, event_id)
- Extended `process_event_reminders` cron with Block 3 for profile subscriptions

**How it works:**
Follower enables via `update_follow_event_subscription(user, profile, enabled, minutes)`. Notification timing:
- `minutes = NULL` → notify exactly at event start time
- `minutes = 5/10/15` → notify X minutes before event start

Applies to ALL events (recurring + non-recurring) on profile. Each occurrence fires separately. Manual event reminders suppress subscriptions for same event (manual takes precedence). Settings persist across unfollow/re-follow.

**Migration required:**
```sql
ALTER TABLE public.follows
  ADD COLUMN IF NOT EXISTS event_notification_enabled boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS event_notification_minutes int NULL;

CREATE TABLE IF NOT EXISTS public.follow_event_subscription_dispatches (
    user_id     uuid        NOT NULL REFERENCES public.users(id)        ON DELETE CASCADE,
    event_id    uuid        NOT NULL REFERENCES public.event_mst(event_id) ON DELETE CASCADE,
    notified_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, event_id)
);
```
