### [2026-06-30] | FEAT | Introduce create_event_v3 — event_end_time is now required #enhancement

**New version:** `create_event_v3` (`POST /rpc/create_event_v3`)
**Deprecated:** `create_event_v2` (`POST /rpc/create_event_v2`) — kept in file as v2.0 deprecated

**Business rule changed:**
`p_event_end_time` is now a **required field** when a creator creates an event. Omitting it returns a validation error. The previous auto-fill behavior (`start_time + 2 hours`) has been removed.

**Why:**
The auto-fill default masked missing data and led to events with incorrect duration. Requiring the creator to explicitly set an end time ensures the duration is intentional and accurate — which matters for conflict detection, LIVE status, and calendar display.

**v3.0 vs v2.0 behavior:**

| Scenario | v3.0 (`create_event_v3`) | v2.0 (`create_event_v2`) |
|----------|--------------------------|--------------------------|
| `p_event_end_time` omitted | ❌ Returns error: "Event end time is required" | Defaults to `p_event_time + 2 hours` |
| `p_event_end_time` provided | ✅ Used as-is | ✅ Used as-is |
| `p_event_end_time = p_event_time` | ❌ Returns error: "cannot be the same as start time" | ❌ Same error |
| Cross-midnight (23:00 → 01:00) | ✅ Handled | ✅ Handled |

**What changed inside the function (v2 → v3):**

```sql
-- REMOVED (auto-fill logic from v2):
IF p_event_end_time IS NULL THEN
    p_event_end_time := p_event_time + (get_config('default_event_duration_hours', '2') || ' hours')::INTERVAL;
END IF;

-- ADDED (required guard in v3):
IF p_event_end_time IS NULL THEN
    RETURN json_build_object('status', false, 'message', 'Event end time is required');
END IF;
```

Parameter signature change:
```sql
-- v2: optional
p_event_end_time  time  DEFAULT null,

-- v3: required (no default)
p_event_end_time  time,
```

End time equality check simplified (null already rejected above, so IS NOT NULL guard removed):
```sql
-- v2:
IF p_event_end_time IS NOT NULL AND p_event_end_time = p_event_time THEN

-- v3:
IF p_event_end_time = p_event_time THEN
```

Everything else is identical to v2 — same recurring logic, same collaborator invites, same platform handling.

**Endpoints:**

- v2.0 (Deprecated): `POST /rpc/create_event_v2`
- v1.0 (Deprecated): `POST /rpc/create_event`
