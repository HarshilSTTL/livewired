**Changes:**

**1. Cross-midnight end_time support**
- `event_end_time < event_time` is now valid and means the event ends on the next day (e.g. start 23:00, end 02:00 → ends at 02:00 the following day).
- All end_time timestamp calculations now use a `CASE WHEN event_end_time < event_time THEN (event_date + 1) ELSE event_date END` expression for the date part.
- Applies to: `get_event_list`, `get_event_by_id`, `get_profile_events`.

**2. Validation change in create_event / update_event**
- Old rule: `end_time <= start_time` → error ("must be after event time").
- New rule: `end_time = start_time` → error ("cannot be the same as event start time"). Values less than start time are allowed (cross-midnight).

**3. Live list criteria change in get_event_list**
- Old: `livestream = true AND event_end_time IS NOT NULL AND started AND not ended`.
- New: `event_end_time IS NOT NULL AND started AND not ended`.
- Any event type (livestream, video, or plain) that has an end_time and is currently active now appears in the Live list.
- The `livestream` flag no longer controls Live placement.

**4. Today list exclusion updated**
- The "exclude events that are currently Live" guard in the Today section no longer checks `livestream = true`, consistent with the Live list change above.

**Changes:**

**1. New table: `event_collaborators`**
- Tracks collaborator invites per event with soft delete + partial unique index
- Status flow: `pending → accepted | declined`; soft-deleted rows can be reactivated on re-invite
- Columns: `id`, `event_id`, `profile_id`, `invited_by`, `status`, `invited_at`, `responded_at`, `updated_at`, `is_deleted`, `deleted_at`

**2. New column: `event_mst.is_collaborative`** (`boolean NOT NULL DEFAULT false`)
- Must be `true` to enable invites on an event

**3. New SPs: `invite_collaborator`, `respond_collaborator_invite`, `remove_collaborator`**
- Invite: validates owner, checks 5-collaborator limit, reactivates soft-deleted rows, notifies invitee
- Respond: accept/decline with race-condition-safe limit recheck, notifies owner
- Remove: owner-only soft delete; removed profile can be re-invited

**4. Ownership extended in `update_event` and `delete_event`** *(corrected in [[updates/2026-05-07.md]])*
- ~~Accepted (non-deleted) collaborators have full permissions: update, postpone, delete~~
- ~~Ownership check: owner OR collaborator via `COALESCE(parent_event_id, event_id)` resolution~~
- **Correction (2026-05-07):** Both SPs were reverted to **owner-only**. Collaborators are read-only — they have no update, delete, or postpone permissions.

**5. Collaborative events in read SPs**
- `get_event_list`: shows collaborative events to the collaborator user regardless of follow status
- `get_profile_events`: includes events where the profile is an accepted collaborator
- `get_event_by_id`: returns `is_collaborative` flag and `collaborators` array (all non-deleted invites)
