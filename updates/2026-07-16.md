### [2026-07-16] | FIX (infra) | event_reminders UNIQUE constraint never actually migrated in Supabase

**What happened:**
`set_event_reminder` failed at runtime with:
```
{status: false, message: "Something went wrong", error: "there is no unique or exclusion constraint matching the ON CONFLICT specification"}
```

**Root cause:**
The `event_reminders` table's `UNIQUE` constraint change (from `(user_id, event_id)` to `(user_id, event_id, reminder_minutes)`) was written into the docs on 2026-07-13 as part of the multi-reminder feature, but the actual `ALTER TABLE` migration was never run against the live Supabase database. The DB still had the old constraint `event_reminders_user_id_event_id_key UNIQUE (user_id, event_id)`, which doesn't match the `ON CONFLICT (user_id, event_id, reminder_minutes)` clause in `set_event_reminder`, causing Postgres to reject the query outright.

**Fix — run in Supabase SQL editor:**
```sql
ALTER TABLE public.event_reminders DROP CONSTRAINT event_reminders_user_id_event_id_key;

ALTER TABLE public.event_reminders
    ADD CONSTRAINT event_reminders_user_id_event_id_reminder_minutes_key
    UNIQUE (user_id, event_id, reminder_minutes);

NOTIFY pgrst, 'reload schema';
```

**Verified via:** `pg_constraint` lookup confirmed the exact stale constraint name/definition before dropping it. Migration ran successfully with no conflicting duplicate rows.

**Status:** Constraint migrated. `set_event_reminder` retest pending confirmation from Harshil (should now succeed; awaiting result for a second reminder time + a `get_event_reminder` check to confirm the `reminders: [...]` array returns both).

**Files:** No doc changes — `schema/tables/14_event_reminders.md` already documented this migration correctly on 2026-07-13; this entry just records that it was actually *applied* to the live database today.
