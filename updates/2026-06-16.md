### [2026-06-16] | FEAT | Create update_event_v2 — collaborator sync behavior #enhancement

**Problem:**
`update_event` (v1) had append-only collaborator logic:
- Passing `[]` did nothing — no way to remove collaborators via `update_event`
- Had to call the separate `remove_collaborator` API to remove anyone
- No way to do a "set this as the exact collaborator list" in one call

**Solution:**
Created `update_event_v2` where `p_collaborator_ids` becomes a **sync** operation — the passed list is the desired final state.

**Sync behavior:**

| Current Collaborators | Payload | Result |
|----------------------|---------|--------|
| (1, 2, 3) | `null` | No change — don't touch collaborators |
| (1, 2, 3) | `[]` | Remove 1, 2, 3 — remove all |
| (1, 2, 3) | `[1,2,3]` | No change |
| (1, 2, 3) | `[1,2]` | Remove 3 |
| (1, 2, 3) | `[1,2,3,4]` | Add 4 |
| (1, 2, 3) | `[2,3,4]` | Remove 1, add 4 |

**What changed inside the function (v1 → v2):**

1. Intent flag changed:
```sql
-- v1 (append-only — triggers only when array has items)
v_update_collabs := p_collaborator_ids IS NOT NULL
                    AND COALESCE(array_length(p_collaborator_ids, 1), 0) > 0;

-- v2 (sync — triggers on any non-null, including empty array)
v_sync_collabs := p_collaborator_ids IS NOT NULL;
```

2. New Step 1 — soft-delete collaborators not in the list:
```sql
UPDATE event_collaborators
SET is_deleted = true,
    deleted_at = now(),
    updated_at = now()
WHERE event_id   = v_target_parent_id
  AND is_deleted = false
  AND (
      COALESCE(array_length(p_collaborator_ids, 1), 0) = 0  -- [] = remove all
      OR profile_id != ALL(p_collaborator_ids)               -- not in new list = remove
  );
```

3. Step 2 — invite new collaborators (same as v1, unchanged):
- Skips already-active collaborators (no re-invite, no notification)
- Re-invites previously soft-deleted collaborators
- Sends notification only to newly invited profiles
- Max 5 accepted collaborators rule still enforced

**Everything else is identical to v1:**
- Scope handling (`all` / `this`)
- Platform sync logic
- Recurring rule + regen
- `is_collaborative` flag handling
- Ownership check

**Endpoints:**
- v2.0 (Current): `POST /rpc/update_event_v2`
- v1.0 (Deprecated): `POST /rpc/update_event`
