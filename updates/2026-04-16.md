### [2026-04-16 02:00] | FIX | google_auth — reactivate soft-deleted account on Google re-login instead of returning stale UUID

**Problem:**
After account deletion, `google_auth` looked up the email with no `is_deleted` filter:
```sql
SELECT id INTO v_user_id FROM users WHERE email = p_email
```
When the user logged in again via Google OAuth, Supabase Auth created a new `auth.users` row (new UUID). But `google_auth` found the old soft-deleted `public.users` row and returned the old UUID as the active `user_id`. The app then used this dead UUID for all subsequent calls including `delete_account`, which returned `"User not found"` because `is_deleted` was already `true` on that row.

**Fix:**
Added `is_deleted` to the SELECT and implemented 3-way logic:
- `is_deleted = false` → return existing `user_id` (normal login)
- `is_deleted = true` → reactivate the row (`is_deleted = false`, `deleted_at = null`) and return the same `user_id`
- Not found → INSERT new row (new Google signup)

Also added `v_is_deleted boolean` to DECLARE and changed email match to case-insensitive (`lower()`).

### [2026-04-16 01:00] | FIX | signup + register — fix "Email already exists" error after account deletion

**Problem:**
Both `signup` and `register` checked for duplicate email with no `is_deleted` filter:
```sql
IF EXISTS (SELECT 1 FROM users WHERE email = p_email) THEN
    RETURN 'Email already exists';
```
After `delete_account`, the `public.users` row remains (`is_deleted = true`). This caused the SP to incorrectly block re-registration with the same email even though the account was deleted.

**Fix:**
Replaced the simple `EXISTS` check with a `SELECT id, is_deleted` lookup:
- `is_deleted = false` → block: return `"Email already exists"` (active account)
- `is_deleted = true` → reactivate: UPDATE the row with new password/username, reset `is_deleted = false`, `deleted_at = null`
- Not found → INSERT brand new row

This prevents duplicate email rows in `public.users` while allowing clean re-registration after deletion.

### [2026-04-16 00:00] | FIX | delete_account — hard delete from auth.users to fix Google re-login and email re-registration bugs

**Problem:**
Two bugs existed due to `delete_account` only soft-deleting `public.users` while leaving `auth.users` untouched:

1. **Google OAuth re-login after deletion** — Supabase Auth still had the `auth.users` row, so Google OAuth re-authenticated the deleted user silently. On a second delete attempt the SP returned `"User not found"`.
2. **Email re-registration after deletion** — `auth.users` still held the email, so Supabase returned `"User already exists"` when the user tried to sign up again.

**Fix:**
Added `DELETE FROM auth.users WHERE id = p_user_id` as the final step in `delete_account`, after the soft-delete updates on `public.users`.

**Strategy — hybrid delete:**
- `public.users`, `creator_profiles`, `event_mst` → **soft delete** (audit trail retained)
- `auth.users` → **hard delete** (email/OAuth identity fully erased)

This allows the user to re-register with the same email or Google account as a brand new user, and prevents silent OAuth re-authentication of a deleted account.
