### [2026-04-16 03:00] | FIX | Full delete-account flow тАФ email anonymization ensures completely fresh account on re-registration

**Problem:**
Previous fixes (reactivation in `google_auth`, `signup`, `register`) returned the **same UUID** as the deleted account. Because all old profiles and events are linked to that UUID, old data was still visible after re-registration. The correct behavior is: re-registration after deletion = brand new UUID = zero old data.

**Root cause:**
The `email` column in `public.users` is `UNIQUE`. After soft deletion, the real email still occupied that slot, so re-registration either blocked with "Email already exists" or was forced to reactivate the old row (same UUID, same data).

**Fix тАФ email anonymization on deletion:**
`delete_account` now anonymizes the email at deletion time:
```sql
email = 'deleted_' || p_user_id::text || '@deleted.invalid'
```
This frees the UNIQUE slot. Re-registration always INSERTs a brand new row with a new UUID тАФ no old profiles, events, or any data attached.

**Changes to `google_auth`, `signup`, `register`:**
Reverted the reactivation logic added in the previous two fixes. All three SPs are back to simple 2-way logic:
- Email found with `is_deleted = false` тЖТ active account exists, block or return user_id
- Email not found тЖТ INSERT fresh row (deleted account's email is anonymized, so real email is always "not found")

**Full flow after this fix:**
```
Delete account тЖТ email anonymized in public.users + auth.users hard deleted
Re-register / Google re-login тЖТ email not found тЖТ brand new UUID inserted тЖТ fresh account
```

### [2026-04-16 02:00] | FIX | google_auth тАФ reactivate soft-deleted account on Google re-login instead of returning stale UUID

**Problem:**
After account deletion, `google_auth` looked up the email with no `is_deleted` filter:
```sql
SELECT id INTO v_user_id FROM users WHERE email = p_email
```
When the user logged in again via Google OAuth, Supabase Auth created a new `auth.users` row (new UUID). But `google_auth` found the old soft-deleted `public.users` row and returned the old UUID as the active `user_id`. The app then used this dead UUID for all subsequent calls including `delete_account`, which returned `"User not found"` because `is_deleted` was already `true` on that row.

**Fix:**
Added `is_deleted` to the SELECT and implemented 3-way logic:
- `is_deleted = false` тЖТ return existing `user_id` (normal login)
- `is_deleted = true` тЖТ reactivate the row (`is_deleted = false`, `deleted_at = null`) and return the same `user_id`
- Not found тЖТ INSERT new row (new Google signup)

Also added `v_is_deleted boolean` to DECLARE and changed email match to case-insensitive (`lower()`).

- [[functions/auth/google_auth.md]]
- [[docs/api/auth/google_auth.md]]

### [2026-04-16 01:00] | FIX | signup + register тАФ fix "Email already exists" error after account deletion

**Problem:**
Both `signup` and `register` checked for duplicate email with no `is_deleted` filter:
```sql
IF EXISTS (SELECT 1 FROM users WHERE email = p_email) THEN
    RETURN 'Email already exists';
```
After `delete_account`, the `public.users` row remains (`is_deleted = true`). This caused the SP to incorrectly block re-registration with the same email even though the account was deleted.

**Fix:**
Replaced the simple `EXISTS` check with a `SELECT id, is_deleted` lookup:
- `is_deleted = false` тЖТ block: return `"Email already exists"` (active account)
- `is_deleted = true` тЖТ reactivate: UPDATE the row with new password/username, reset `is_deleted = false`, `deleted_at = null`
- Not found тЖТ INSERT brand new row

This prevents duplicate email rows in `public.users` while allowing clean re-registration after deletion.

### [2026-04-16 00:00] | FIX | delete_account тАФ hard delete from auth.users to fix Google re-login and email re-registration bugs

**Problem:**
Two bugs existed due to `delete_account` only soft-deleting `public.users` while leaving `auth.users` untouched:

1. **Google OAuth re-login after deletion** тАФ Supabase Auth still had the `auth.users` row, so Google OAuth re-authenticated the deleted user silently. On a second delete attempt the SP returned `"User not found"`.
2. **Email re-registration after deletion** тАФ `auth.users` still held the email, so Supabase returned `"User already exists"` when the user tried to sign up again.

**Fix:**
Added `DELETE FROM auth.users WHERE id = p_user_id` as the final step in `delete_account`, after the soft-delete updates on `public.users`.

**Strategy тАФ hybrid delete:**
- `public.users`, `creator_profiles`, `event_mst` тЖТ **soft delete** (audit trail retained)
- `auth.users` тЖТ **hard delete** (email/OAuth identity fully erased)

This allows the user to re-register with the same email or Google account as a brand new user, and prevents silent OAuth re-authentication of a deleted account.
