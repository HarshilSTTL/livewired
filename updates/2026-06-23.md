### [2026-06-23] | FEAT | Introduce create_event_v2 — default end time to 2 hours #enhancement

**New version:** `create_event_v2` (`POST /rpc/create_event_v2`)
**Deprecated:** `create_event` (`POST /rpc/create_event`) — kept in file as v1.0

**Business rule added:**
If a creator does not provide `p_event_end_time`, `create_event_v2` now automatically defaults it to **2 hours after** `p_event_time`.

**Why:**
Events without an end time had no duration context — they couldn't show a "LIVE" status and produced less useful conflict-check results. Defaulting to 2 hours gives every event a sensible duration without forcing the creator to always specify one.

**v2.0 vs v1.0 behavior:**

| Scenario | v2.0 (`create_event_v2`) | v1.0 (`create_event`) |
|----------|--------------------------|----------------------|
| `p_event_end_time` omitted | Defaults to `p_event_time + 2 hours` | Stored as `NULL` |
| `p_event_end_time` provided | Used as-is | Used as-is |
| Cross-midnight wrap (23:00 → 01:00) | ✅ Handled | ✅ Handled |

**Cross-midnight note:**
PostgreSQL `TIME + INTERVAL '2 hours'` wraps correctly past midnight (e.g. `23:00 → 01:00`).
The existing cross-midnight logic in `get_profile_events` and `get_event_list` already handles `event_end_time < event_time` — no additional changes needed there.

**What changed inside the function (v1 → v2):**

```sql
-- Added after end-time validation, before INSERT into event_mst
IF p_event_end_time IS NULL THEN
    p_event_end_time := p_event_time + INTERVAL '2 hours';
END IF;
```

Everything else is identical to v1 — same params, same recurring logic, same collaborator invites.

**Endpoints:**
- v2.0 (Current): `POST /rpc/create_event_v2`
- v1.0 (Deprecated): `POST /rpc/create_event`

**Files Modified:**
- [[functions/events/create_event.md]] — Added Version History section; `create_event_v2` at top (current), original `create_event` moved to bottom as v1.0 deprecated
- [[docs/api/events/create_event.md]] — Added Versions table, v2.0 vs v1.0 comparison table; updated `p_event_end_time` default to `start + 2h`

### [2026-06-23] | FEAT | Introduce system_config table for app-wide settings #enhancement

**New table:** `system_config` — centralized configuration for all changeable app settings

**Why:** Previously, settings were hardcoded in SQL functions (e.g., "max 5 collaborators", "token expires in 24 hours", "default event duration 2 hours"). Now admins can change these without code changes.

**What's in system_config:**

| Category | Settings | Count |
|----------|----------|-------|
| Auth | Email verification expiry, password length, login attempts, lockout duration, resend cooldown | 7 |
| Event | Default event duration, max collaborators, recurring max months, conflict check | 4 |
| Platform | Default platforms, featured platforms, stream URL validation | 3 |
| Notification | Recurring expiry notification days, retention days | 2 |
| General | App name, username length (min/max) | 3 |

**How to change settings:**
- No admin API — directly update the `system_config` table in Supabase SQL Editor
- Example: `UPDATE system_config SET config_value = '10' WHERE config_key = 'max_collaborators_per_event';`
- SPs read fresh values on every call via the `get_config()` helper function

**Helper function:**
- `get_config(key, default)` — Used internally by other SPs to read config values from the table

**Updated SPs to use config:**
- `create_event` (v1.0) — reads `default_event_duration_hours` and `max_collaborators_per_event` from config
- `create_event_v2` — reads same config values instead of hardcoded defaults
- `signup` — will read `email_verification_expiry_hours` when email verification is added

**18 default config values inserted:**
- 7 auth settings (verification expiry, password rules, login lockout)
- 4 event settings (default duration, max collaborators, recurring generation, conflict check)
- 3 platform settings (defaults, featured, URL validation)
- 2 notification settings (expiry notification, retention)
- 2 general settings (app name, username length min/max)
