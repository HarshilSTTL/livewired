### [2026-05-11 15:00] | FEAT + TABLE + DOCS + CRON | Profile owner event notifications

**What changed:**
New system for profile owners to receive notifications for any event on their profiles.
Separate from follower reminders — this is for the profile owner themselves.

**What added:**
- New table `profile_event_notifications` (user_id, profile_id, notification_enabled, notification_type, reminder_minutes)
- New SP `update_profile_event_notification` to configure notifications per profile
- Extended `process_event_reminders` cron with profile owner notification block

**How it works:**
Profile owner configures via `update_profile_event_notification(p_user_id, p_profile_id, p_notification_enabled, p_notification_type, p_reminder_minutes)`.
Three notification types:
- `'before_event'` — fires X minutes before event starts (1–1440 min configurable)
- `'on_event_start'` — fires when event actually starts
- `'both'` — fires both before AND at start time

Applies to ALL events on the profile. Each recurring occurrence fires separately.

**Migration required:**
```sql
CREATE TABLE IF NOT EXISTS public.profile_event_notifications (
    id                    uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id               uuid        NOT NULL REFERENCES public.users(id)            ON DELETE CASCADE,
    profile_id            uuid        NOT NULL REFERENCES public.creator_profiles(id) ON DELETE CASCADE,
    notification_enabled  boolean     NOT NULL DEFAULT false,
    notification_type     text        NOT NULL DEFAULT 'before_event',
    reminder_minutes      int         NOT NULL DEFAULT 10,
    created_at            timestamptz DEFAULT now(),
    updated_at            timestamptz DEFAULT now()
);

CREATE UNIQUE INDEX IF NOT EXISTS uq_profile_event_notifications
    ON public.profile_event_notifications (user_id, profile_id);
```

### [2026-05-11 13:00] | FEAT + DOCS | `postpone_recurring_occurrence` — move single occurrence to new date/time

**What changed:**
New SP to reschedule a single occurrence in a recurring series without affecting parent or other occurrences.

**Params:**
- `p_event_id uuid` — required; the child occurrence to postpone
- `p_user_id uuid` — required; event owner auth check
- `p_new_date date` — required; must be >= today
- `p_new_time time` — optional; if omitted, keeps existing time

**Logic:**
- Null guards on p_event_id, p_user_id, p_new_date
- Locate event; verify parent_event_id IS NOT NULL (child occurrence only)
- Ownership check: caller must be event owner
- Validate new_date >= CURRENT_DATE (no past dates)
- v_final_time = COALESCE(p_new_time, existing time)
- UPDATE event_mst SET event_date = p_new_date, event_time = v_final_time

**Error cases:** `p_event_id, p_user_id, and p_new_date are required` · `Event not found` · `You do not have permission to modify this event` · `Cannot postpone a parent event or non-recurring event` · `New date cannot be in the past`

### [2026-05-11 11:00] | FEAT + TABLE + DOCS | Per-follower auto-reminder system (Tasks #12 + #13)

**What changed:**
YouTube-style bell-icon notification system. Followers opt in to automatic event reminders on a per-profile basis, configurable from 1–1440 minutes before each event. Applies to all events on the profile including every occurrence of a recurring series.

**What added:**
- Two columns to `follows` table: `reminder_enabled` (boolean, default false) and `reminder_minutes` (int, default 10)
- New table `follow_reminder_dispatches` (user_id, event_id, notified_at) for exactly-once delivery ledger
- New SP `update_follow_reminder` to toggle bell + set minutes per-(user, profile)
- Extended `process_event_reminders` cron with follow-level reminder block

**How it works:**
- Follower sets bell via `update_follow_reminder(p_user_id, p_profile_id, p_reminder_enabled, p_reminder_minutes)`
- Cron checks `follows.reminder_enabled = true` and fires notification (data.type = 'follow_reminder') X minutes before each event
- Manual `event_reminders` rows suppress follow-level reminders for that event (manual wins)
- Composite PK on dispatches table with ON CONFLICT DO NOTHING guarantees exactly-once delivery per (user, event)
- Settings persist across unfollow/re-follow cycles

**Recurring series handling:**
Each child occurrence is a separate `event_mst` row, so the same per-follower setting fires once per occurrence.
Example: Monday meetings for 3 months → 13 reminder notifications (one per occurrence).

**Migration required:**
```sql
ALTER TABLE public.follows
  ADD COLUMN IF NOT EXISTS reminder_enabled boolean NOT NULL DEFAULT false,
  ADD COLUMN IF NOT EXISTS reminder_minutes int     NOT NULL DEFAULT 10;

CREATE TABLE IF NOT EXISTS public.follow_reminder_dispatches (
    user_id     uuid        NOT NULL REFERENCES public.users(id)        ON DELETE CASCADE,
    event_id    uuid        NOT NULL REFERENCES public.event_mst(event_id) ON DELETE CASCADE,
    notified_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, event_id)
);
```

### [2026-05-11 10:00] | FIX + DOCS | `update_event` — auto-null `recurring_interval` for first/last types

**What changed:**
Recurring rule merge now forces `v_rec_interval := NULL` when resolved type is 'first' or 'last'.
Removed the "must be null for first/last" guard — now auto-handles the constraint.

**Root cause:**
COALESCE pattern `v_rec_interval := COALESCE(p_recurring_interval, v_rec_interval)` leaked the stored weekly interval when user omitted p_recurring_interval for a 'first'/'last' type change.

**Fix:**
Branch on resolved type:
- If type = 'weekly': COALESCE as normal
- If type = 'first' or 'last': force NULL regardless of stored value

### [2026-05-11 09:00] | FIX + DOCS | `update_event` — auto-demote `'this'` for non-recurring / parent rows

**What changed:**
Added automatic scope demotion: when p_scope = 'this' is passed for a non-recurring event (parent_event_id IS NULL), demote to 'all' instead of erroring.

**Root cause:**
Client form state can pass scope='this' for a non-recurring event by accident; erroring created a poor UX.

**Fix:**
After parent resolution, check: `IF v_scope = 'this' AND v_parent_event_id IS NULL THEN v_scope := 'all'`
Removed the "Scope 'this' can only be used on…" guard.

**Behaviour after fix:**
- 'this' scope on parent row → auto-demotes to 'all' (applies to whole series)
- 'this' scope on non-recurring row → auto-demotes to 'all' (applies to single event)
