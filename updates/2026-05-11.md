**Symptoms fixed:**

1. `p_scope='this'` with `p_recurring_days` omitted entirely → returned `Recurring schedule cannot be changed for a single occurrence — use scope 'all'`. Root cause: a stale overload of `update_event` from before `p_scope` was added (2026-05-08) was still in the DB. PostgREST routed to whichever overload matched the JSON keys; the older one had a different default for `p_recurring_days`, so its guard fired even though the new SP wouldn't have. Resolved by the deployment block above + the deploy-time note in the SP.
2. `p_scope='this'` with `p_collaborator_ids: []` from form state → was rejected by an asymmetric guard (`IS NOT NULL` only) where the collaborator guard was `IS NOT NULL AND array_length > 0`. Now both guards are symmetric and `null` / `[]` are equivalent everywhere in the SP.
3. `p_scope='this'` with non-empty `p_collaborator_ids` (and optionally `p_is_collaborative`) → was rejected with `Collaborator invites cannot be scoped to a single occurrence`. Now auto-routes: the scalar/platform changes apply to the child occurrence; the `is_collaborative` flag is set on the parent (and propagated to all children); the collaborator invites are recorded against the parent series and notifications go out. The combined update returns `Event occurrence updated successfully` with `data.skipped_collaborator_ids` reflecting any invites that were skipped (already-invited, self-invite, cap hit, inactive profile).
4. `p_scope='this'` with `p_is_collaborative` was silently dropped (the child UPDATE didn't include the column). Now treated as series-level and applied to the parent + propagated to children regardless of scope.

**Still rejected (by design):**

- `p_scope='this'` with non-empty `p_recurring_days` — regenerating children from a new recurring rule would DELETE the very row being edited. Use `p_scope='all'` to change the schedule.
