### [2026-07-15] | FEAT + SCHEMA | Track onboarding completion (platform/tag choose screens) #enhancement

**What changed:**
Added `onboarding_completed` flag to `public.users` so the app can tell whether a user has been through the platform-choose / tag-suggestion onboarding screens on first login — including when they tap **Skip**.

**Why:**
There was previously no reliable way to know if a user was "first time." Inferring it from whether `user_preferred_platforms`/`user_interests` had rows breaks the moment a user taps Skip (no rows get written), which would show onboarding again on every subsequent login.

**Schema change:**
```sql
ALTER TABLE public.users ADD COLUMN IF NOT EXISTS onboarding_completed boolean NOT NULL DEFAULT false;
```

**Functions updated/added:**

| Function | Change |
|---|---|
| `get_user` | Response now includes `onboarding_completed` |
| `submit_platform` | Sets `onboarding_completed = true` after saving platforms |
| `submit_tags` | Sets `onboarding_completed = true` after saving tags |
| `skip_onboarding` | New — sets `onboarding_completed = true` with no platform/tag writes, for the Skip button |

### [2026-07-15] | FIX | Duplicate-email crash in handle_new_auth_user trigger

**What changed:**
`handle_new_auth_user`'s mirror INSERT into `public.users` was `ON CONFLICT (id) DO NOTHING` — only guarding against `id` collisions. If a `public.users` row already existed with the same `email` under a **different** `id` (e.g. leftover data from the deprecated `register`/`signup`/`google_auth` RPCs), the INSERT threw an unhandled unique-violation, rolling back the entire `auth.users` insert and failing signup with a `500`.

**Fix:** changed to a bare `ON CONFLICT DO NOTHING` (no column list), which absorbs a violation on either unique constraint. Added a `RAISE WARNING` when the insert is skipped, since that leaves the `auth.users` account with no matching `public.users` row (needs manual reconciliation) instead of failing silently.

**Root cause found via:** Supabase Auth Logs — a real signup attempt for a test account returned `500` with `23505 duplicate key value violates unique constraint "users_email_key"`.

### [2026-07-15] | FIX (infra) | Email verification not sending — custom SMTP required

**What changed:**
Verification emails were not being delivered. Root cause: project was on Supabase's default built-in mailer, which is rate-limited and explicitly "not meant to be used for production apps" — confirmed via the dashboard warning banner. Configured custom SMTP (Gmail relay, `smtp.gmail.com:587`, sender `harshil22kachhadiya@gmail.com`, auth via Google App Password) under Authentication → Emails → SMTP Settings.

**Also cleaned up:** two test accounts (`prajapatimeet1907@gmail.com`, `meetpraja19@gmail.com`) created during diagnosis, including their dependent rows in `user_preferred_platforms`, `user_interests`, and `creator_profiles` (none of which cascade from `public.users` — deletion required explicit per-table cleanup first).

**Note:** this is an infrastructure/dashboard config change, not a schema or function change — no doc files affected beyond the trigger fix above.
