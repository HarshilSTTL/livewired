### [2026-05-04 10:00] | SCHEMA + SP | Recurring event end date default + renewal notification

**Feature:** `recurring_end_date` now defaults to 3 months (was 1 year) when not provided. The computed end date is always stored in `event_recurring`. A new SP `notify_expiring_recurring_events` sends a push notification to the event owner 7 days before the recurring schedule ends, to prompt renewal.

---

#### [[schema/tables/13_event_recurring.md]]

**SCHEMA — new column `renewal_notified_at`**
- **Before:** Table had no way to track whether a renewal notification had been sent. Running any notification job would send duplicate messages on subsequent invocations.
- **After:** Added `renewal_notified_at timestamptz DEFAULT NULL`. Set to `now()` by `notify_expiring_recurring_events` after sending the notification. `NULL` = not yet notified. This is the idempotency guard — the SP filters `WHERE renewal_notified_at IS NULL` so each event is only notified once.
- `update_event` resets `renewal_notified_at = NULL` whenever the recurring schedule is changed, so the owner will be notified again for the new end date.
- Migration SQL:
  ```sql
  ALTER TABLE public.event_recurring ADD COLUMN IF NOT EXISTS renewal_notified_at timestamptz DEFAULT NULL;
  ```

#### [[functions/events/create_event.md]]

**FIX — default recurring end date changed from 1 year → 3 months**
- **Before:**
  ```sql
  v_safe_end := COALESCE(p_recurring_end_date, p_recurring_start_date + INTERVAL '1 year');
  ```
  Generated up to 1 year of child occurrence rows when no end date was provided.
- **After:**
  ```sql
  v_safe_end := COALESCE(p_recurring_end_date, p_recurring_start_date + INTERVAL '3 months');
  ```
  Generates up to 3 months of occurrences by default.

**FIX — always store a concrete `recurring_end_date` in `event_recurring`**
- **Before:** The INSERT into `event_recurring` passed `p_recurring_end_date` directly. If the user omitted it, `recurring_end_date` was stored as `NULL` — the notification SP cannot query a `NULL` end date.
- **After:** The INSERT now passes `v_safe_end` (the already-computed COALESCE result). `recurring_end_date` always contains a real date after creation — either the user-supplied value or `start_date + 3 months`. The `NULL` case is eliminated at write time.
  ```sql
  -- Before
  VALUES (..., p_recurring_end_date, now())

  -- After
  VALUES (..., v_safe_end, now())
  ```

#### [[functions/events/update_event.md]]

**FIX — same 3-month default applied during recurring schedule update**
- **Before:**
  ```sql
  v_safe_end := COALESCE(v_rec_end, v_rec_start + INTERVAL '1 year');
  ```
- **After:**
  ```sql
  v_safe_end := COALESCE(v_rec_end, v_rec_start + INTERVAL '3 months');
  v_rec_end  := v_safe_end;
  ```
  `v_rec_end` is updated to the computed value so the UPDATE statement stores it.

**FIX — store computed end date + reset `renewal_notified_at` on schedule change**
- **Before:** The `UPDATE event_recurring SET ...` block updated only schedule columns. Stored `recurring_end_date` could remain as an old value or NULL.
- **After:** The UPDATE now includes:
  - `recurring_end_date = v_rec_end` — always a concrete date
  - `renewal_notified_at = NULL` — resets the notification flag so the owner will be reminded again for the new end date

#### [[functions/events/notify_expiring_recurring_events.md]] *(new file)*

**SP — `POST /rpc/notify_expiring_recurring_events`**
- No parameters. Intended to be called by a pg_cron daily schedule.
- **Query:** Selects all `event_recurring` rows WHERE:
  - `recurring_end_date BETWEEN CURRENT_DATE AND CURRENT_DATE + INTERVAL '7 days'`
  - `renewal_notified_at IS NULL`
  - `event_mst.is_deleted = false`
  - `creator_profiles.status = 'active'`
- **For each matching row:**
  - Inserts a notification for the event owner:
    - `title = 'Recurring Event Ending Soon'`
    - `body = 'Your recurring event "<title>" ends on <date>. Update the schedule to keep it going.'`
    - `data = { type: 'recurring_renewal', event_id, recurring_end_date }`
  - Sets `renewal_notified_at = now()` to prevent duplicate notifications
- **Returns:** `{ "status": true, "message": "N renewal notification(s) sent" }`
- **Idempotent:** Running multiple times in the same day sends no duplicates.
- **pg_cron schedule (run once in Supabase SQL editor):**
  ```sql
  SELECT cron.schedule(
      'notify-expiring-recurring-events',
      '0 9 * * *',
      $$SELECT notify_expiring_recurring_events()$$
  );
  ```
