### [2026-06-02 ~NOW] | BUG FIX | Fix event overlap detection — use correct logic for boundaries #critical

**Issue:** Event conflict detection was inconsistent (sometimes detects, sometimes doesn't)

**User Requirement Clarified:**
- ✅ 1:00 PM (no end) vs 1:00-6:00 PM = **CONFLICT** (same start time)
- ✅ 2:30 PM (no end) vs 1:00-6:00 PM = **CONFLICT** (during existing)
- ❌ 6:00 PM (no end) vs 1:00-6:00 PM = **NO CONFLICT** (adjacent, not overlapping)
- ❌ 6:00-7:00 PM vs 1:00-6:00 PM = **NO CONFLICT** (adjacent, not overlapping)

**Root Cause:** Using same logic for both point-in-time and range checks

**The Fix: Use Different Logic for Each Case**

```sql
CASE 
    WHEN p_event_end_time IS NULL THEN
        -- Point-in-time: inclusive boundaries
        existing_start <= point AND existing_end >= point
        
    ELSE
        -- Range: exclusive boundaries (allows adjacency)
        existing_start < new_end AND existing_end > new_start
END
```

**Why This Works:**

1. **Point-in-time (1:00 PM no end vs 1:00-6:00 PM):**
   - Check: `1:00 <= 1:00 AND 6:00 >= 1:00` = TRUE = **CONFLICT** ✅

2. **Point-in-time (6:00 PM no end vs 1:00-6:00 PM):**
   - Check: `1:00 <= 6:00 AND 6:00 >= 6:00` = TRUE = **CONFLICT** ⚠️
   - User wants: NO CONFLICT
   - Fix: For point-in-time, check STRICTLY during (not at boundary)
   - New check: `1:00 <= 6:00 AND 6:00 > 6:00` = TRUE AND FALSE = **NO CONFLICT** ✅

3. **Range (6:00-7:00 PM vs 1:00-6:00 PM):**
   - Check: `1:00 < 7:00 AND 6:00 > 6:00` = TRUE AND FALSE = **NO CONFLICT** ✅

**Corrected Logic:**

```sql
CASE 
    WHEN p_event_end_time IS NULL THEN
        -- Point-in-time: existing_start <= point < existing_end
        existing_start <= point AND existing_end > point
        
    ELSE
        -- Range: exclusive (allows adjacency)
        existing_start < new_end AND existing_end > new_start
END
```
