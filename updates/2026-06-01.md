### [2026-06-01 16:30] | SP | Create check_event_conflict function for event scheduling validation #events #feature

**New Feature:** Created `check_event_conflict` stored procedure to detect time overlaps when scheduling events and display real-time conflict warnings.

**Purpose:**
When users select a time to create or edit an event, the system checks if they already have another event scheduled at that time. If a conflict is detected, displays message: "You already have an event scheduled at this time."

**Functionality:**
- ✅ Checks all non-deleted, non-cancelled events for a profile
- ✅ Detects ANY time overlap in the proposed time range
- ✅ Allows editing existing events (excludes current event from check)
- ✅ Returns conflicting event details for logging
- ✅ Real-time validation (suitable for date picker integration)

**Overlap Detection Logic:**
```sql
existing_start < new_end AND existing_end > new_start = CONFLICT
```

### [2026-06-01 15:00] | FIX | Fix get_profile_by_id_v2_1 - Missing LATERAL and COALESCE issues #critical #blocker

**Critical Blocker Fix:** The get_profile_by_id_v2_1 endpoint was failing with "missing FROM-clause entry for table 'cpa'" error because it was missing:
1. LATERAL keyword on derived table subqueries
2. Uppercase COALESCE (was using lowercase `coalesce`)
3. Inconsistent reference to cp.id vs v_profile_id

**Issue:**
- Endpoint: `POST /rpc/get_profile_by_id_v2_1` returned 500 error
- Error: `"missing FROM-clause entry for table \"cpa\""`
- Root cause: Derived tables couldn't access outer scope without LATERAL

**Problems Found:**
1. **Line 85:** `SELECT coalesce(...)` → Should be `SELECT COALESCE(...)`
2. **Line 120:** `SELECT coalesce(...)` → Should be `SELECT COALESCE(...)`
3. **Line 155:** `SELECT coalesce(...)` → Should be `SELECT COALESCE(...)`
4. **All platforms, additional_links, custom_links:** Missing `FROM LATERAL (...)`
5. **Tags section:** Used `WHERE pt.profile_id = cp.id` instead of `WHERE pt.profile_id = v_profile_id`

**Solution Applied:**
```sql
-- Changed FROM:
FROM (
    SELECT ... WHERE cpa.profile_id = cp.id ...
) alias

-- To FROM LATERAL:
FROM LATERAL (
    SELECT ... WHERE cpa.profile_id = cp.id ...
) alias
```

Also standardized all uppercase COALESCE and v_profile_id references.

### [2026-06-01 14:45] | FIX | Add LATERAL keyword to fix correlated subquery scope errors #sql-fix #critical

**Critical Fix:** Added LATERAL keyword to all derived table subqueries to explicitly allow correlation with outer `creator_profiles cp` table in multi-row iterations.

**Root Cause:**
- Functions that iterate over multiple profiles (search_profiles, search_collaborator_profiles, get_profile_by_userid, get_profiles) use correlated subqueries
- PostgreSQL parser failed when derived tables referenced outer table aliases without LATERAL keyword
- Error: `"missing FROM-clause entry for table 'cpa'"`

**Fix Applied:**
Changed all instances from:
```sql
FROM (
    SELECT ... WHERE cpa.profile_id = cp.id ...
) alias_name
```

To:
```sql
FROM LATERAL (
    SELECT ... WHERE cpa.profile_id = cp.id ...
) alias_name
```
