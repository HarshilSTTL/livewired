# `16_follow_reminder_dispatches`

```sql
-- Table: follow_reminder_dispatches
-- Purpose: Exactly-once delivery ledger for follow-level (auto) reminders.
--          The cron INSERTs into this table with ON CONFLICT DO NOTHING; only rows that
--          actually land here produce a notification row, so duplicate cron firings or
--          multiple cron workers can never double-notify a follower.
--          One row per (user_id, event_id) means "this follower has been auto-notified
--          for this specific event"; recurring children each get their own row.
-- Doc: docs/database/tables/16_follow_reminder_dispatches.md

CREATE TABLE IF NOT EXISTS public.follow_reminder_dispatches (
    user_id     uuid        NOT NULL REFERENCES public.users(id)        ON DELETE CASCADE,
    event_id    uuid        NOT NULL REFERENCES public.event_mst(event_id) ON DELETE CASCADE,
    notified_at timestamptz NOT NULL DEFAULT now(),
    PRIMARY KEY (user_id, event_id)
);

-- Cascade rules:
--   user_id  ON DELETE CASCADE → user removed → all their dispatch records go too
--   event_id ON DELETE CASCADE → event removed → its dispatch records go too
--   (Recurring rule change in update_event deletes child rows; their dispatch records
--    are cleared automatically. New children regenerated by the rule update have no
--    prior dispatches and will fire correctly when their event_time approaches.)

-- Not tracked here:
--   • Manual per-event reminders set by a follower — those live in event_reminders with
--     their own is_notified flag.
--   • Owner / collaborator notifications — those are one-shot inserts at create time.
```
